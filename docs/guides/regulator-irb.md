# Regulator / IRB Guide

**National MCP-PAI Oncology Trials Standard**
**Audience**: FDA Inspectors, State Regulators, Institutional Review Boards, Quality Assurance
**Version**: 0.1.0

---

## 1. Introduction

This guide assists regulatory inspectors, IRB reviewers, and quality assurance
personnel in understanding, evaluating, and inspecting deployments of the National
MCP-PAI Oncology Trials Standard. It covers the evidence package structure, conformance
artifact interpretation, audit replay format, change control procedures, inspection
readiness, and regulatory submission guidance.

The standard operates within a layered regulatory framework:

| Layer | Scope | Reference |
|-------|-------|-----------|
| Federal | 21 CFR Part 11, HIPAA, ICH-GCP E6(R2) | `profiles/country-us-fda.md`, `regulatory/` |
| State | Jurisdiction-specific overlays | `profiles/state-us-ca.md`, `profiles/state-us-ny.md` |
| Institutional | IRB and site-specific policies | `regulatory/IRB_SITE_POLICY_TEMPLATE.md` |

## 2. Evidence Package Structure and Contents

### 2.1 Evidence Pack Overview

The certification evidence pack is generated by `tools/certification/evidence_pack.py`
and contains all artifacts necessary to demonstrate conformance. Each evidence pack
is a self-contained, cryptographically sealed bundle.

### 2.2 Evidence Pack Structure

```
evidence-pack-{site_id}-{timestamp}/
  manifest.json              # Pack metadata, SHA-256 hashes of all artifacts
  conformance/
    test_results.json        # Conformance test pass/fail results
    test_log.txt             # Full test execution log
    schema_validation.json   # Schema validation results for all 13 schemas
  audit/
    chain_status.json        # ledger_chain_status output
    chain_verification.json  # ledger_verify output (full chain)
    sample_records.json      # Representative audit records
  provenance/
    source_registry.json     # Registered data sources
    lineage_samples.json     # Representative lineage chains
    integrity_checks.json    # provenance_verify_integrity results
  security/
    token_lifecycle.json     # Token issuance, validation, revocation tests
    chain_integrity.json     # Tamper detection test results
    ssrf_prevention.json     # SSRF prevention test results
  safety/
    gate_decisions.json      # Safety gate evaluation records
    robot_registry.json      # Registered robots and USL scores
    estop_tests.json         # E-stop test results
  environment/
    site_profile.yaml        # Site capability profile
    server_versions.json     # Server version information
    deployment_config.json   # Deployment configuration snapshot
```

### 2.3 Evidence Artifact Properties

Each artifact in the evidence pack contains (per `tools/certification/evidence_pack.py`):

| Property | Description |
|----------|-------------|
| `name` | Human-readable artifact name |
| `artifact_type` | Classification (test_results, schema_validation, chain_verification, etc.) |
| `content` | Artifact content (serializable) |
| `sha256` | SHA-256 hash computed over sorted JSON serialization |

### 2.4 Manifest Structure

The manifest provides a tamper-evident index of all artifacts:

```json
{
  "pack_id": "evp-site-us-ca-stanford-20260308T120000Z",
  "timestamp": "2026-03-08T12:00:00Z",
  "site_id": "site-us-ca-stanford",
  "profile": "robot-assisted-procedure",
  "level": 5,
  "artifacts": [
    {
      "name": "conformance_test_results",
      "artifact_type": "test_results",
      "sha256": "a3f8e2d1b4c5..."
    }
  ],
  "pack_hash": "aggregate SHA-256 of all artifact hashes"
}
```

## 3. Conformance Artifact Interpretation

### 3.1 Conformance Levels

The standard defines five conformance levels (per `spec/conformance.md`). Each level
builds on the previous:

| Level | Name | Required Servers | Profile Reference |
|-------|------|-----------------|-------------------|
| 1 | Core | AuthZ, Ledger | `profiles/base-profile.md` |
| 2 | Clinical Read | + FHIR | `profiles/clinical-read.md` |
| 3 | Imaging | + DICOM | `profiles/imaging-guided-oncology.md` |
| 4 | Federated Site | + Provenance | `profiles/multi-site-federated.md` |
| 5 | Robot-Assisted Procedure | All 5 servers | `profiles/robot-assisted-procedure.md` |

### 3.2 Interpreting Test Results

Conformance test results use the following status indicators:

| Status | Meaning | Regulatory Implication |
|--------|---------|----------------------|
| PASS | Requirement satisfied | Conformant |
| FAIL | Requirement not satisfied | Non-conformant; remediation required |
| SKIP | Test not applicable (lower level) | Acceptable if below declared level |
| ERROR | Test execution error | Inconclusive; re-test required |

### 3.3 Test Categories

| Category | Directory | What It Validates |
|----------|-----------|-------------------|
| Positive (core) | `conformance/positive/test_core_conformance.py` | AuthZ RBAC and ledger chain operations |
| Positive (clinical) | `conformance/positive/test_clinical_read_conformance.py` | FHIR de-identification and access controls |
| Positive (imaging) | `conformance/positive/test_imaging_conformance.py` | DICOM role-based access and privacy |
| Blackbox (per server) | `conformance/blackbox/test_*_conformance.py` | Each server's tool contracts |
| Cross-server workflow | `conformance/blackbox/test_cross_server_workflow.py` | End-to-end multi-server scenarios |
| Security | `conformance/security/` | Token lifecycle, chain integrity, SSRF |
| Adversarial | `conformance/adversarial/` | Attack resistance (replay, PHI leakage, bypass) |
| Negative | `conformance/negative/` | Proper rejection of invalid/unauthorized inputs |
| Interoperability | `conformance/interoperability/` | Cross-server trace and schema validation |

## 4. Audit Replay Package Format

### 4.1 Audit Chain Architecture

The audit ledger (`servers/trialmcp_ledger/`) uses a SHA-256 hash chain. Each record
contains:

| Field | Description |
|-------|-------------|
| `audit_id` | Unique record identifier |
| `hash` | SHA-256 hash of this record |
| `previous_hash` | SHA-256 hash of the preceding record |
| `timestamp` | ISO 8601 UTC timestamp |
| `server` | Originating MCP server |
| `tool` | Tool that was invoked |
| `caller` | Actor who made the call |
| `parameters` | De-identified tool call parameters |
| `result_summary` | Outcome description |

The genesis record uses `"0" * 64` (64 zero characters) as its `previous_hash`.

### 4.2 Chain Verification

To verify audit chain integrity:

```python
# Full chain verification
result = ledger_verify()
# Returns:
# {
#   "valid": true/false,
#   "records_checked": integer,
#   "first_invalid_index": null (if valid) or integer (if broken),
#   "genesis_valid": true/false
# }
```

A valid chain satisfies:
1. The genesis record has `previous_hash == "0" * 64`
2. Each subsequent record's `previous_hash` equals the `hash` of the prior record
3. Each record's `hash` can be independently recomputed from its contents

### 4.3 Replay Trace Format

The `ledger_replay` tool generates sequential traces suitable for compliance review:

```json
{
  "trace": [
    {
      "audit_id": "aud-001",
      "timestamp": "2026-03-01T09:00:00Z",
      "server": "trialmcp-authz",
      "tool": "authz_evaluate",
      "caller": "trial-coordinator-jane",
      "parameters": {"role": "robot_agent", "server": "trialmcp-fhir", "tool": "fhir_read"},
      "result_summary": "ALLOW",
      "hash": "a3f8...",
      "previous_hash": "0000..."
    },
    {
      "audit_id": "aud-002",
      "timestamp": "2026-03-01T09:00:01Z",
      "server": "trialmcp-fhir",
      "tool": "fhir_read",
      "caller": "robot-da-vinci-001",
      "parameters": {"resource_type": "Patient", "resource_id": "hmac_a3f8e2d1b4c5"},
      "result_summary": "De-identified Patient resource returned",
      "hash": "b4c5...",
      "previous_hash": "a3f8..."
    }
  ],
  "total": 2,
  "duration": "PT1S"
}
```

### 4.4 21 CFR Part 11 Audit Trail Mapping

The MCP audit chain satisfies 21 CFR Part 11 Section 11.10(e) requirements:

| Part 11 Requirement | MCP Implementation |
|---------------------|-------------------|
| Computer-generated, time-stamped audit trail | ISO 8601 UTC `timestamp` on every record |
| Record date/time of operator entry | `timestamp` and `caller` fields |
| Record identity of operator | `caller` field with authenticated identity |
| Record changes to electronic records | `parameters` and `result_summary` fields |
| Previously recorded information not obscured | Append-only hash chain; records immutable |
| Audit trail retained for required period | Site retention policy per predicate rule |
| Available for FDA review and copying | `ledger_replay` export; human-readable format |

## 5. Change Control and Validation Package

### 5.1 Change Categories

Per `governance/DECISION_PROCESS.md`, changes follow a structured process:

| Category | Scope | Review Period | Approvals |
|----------|-------|---------------|-----------|
| Editorial (Patch) | Typos, formatting | 7 days | 1 maintainer |
| Minor Enhancement | New tools, parameters | 14 days | 2 maintainers |
| Breaking Change (Major) | Contract changes, security model | 30 days | All maintainers |

### 5.2 Validation Requirements

Per 21 CFR Part 11 Section 11.10(a) (see `profiles/country-us-fda.md` Section 3.1):

| Validation Activity | Evidence |
|---------------------|----------|
| System validation | Conformance test results (all declared levels) |
| Schema validation | JSON Schema draft 2020-12 validation for all inputs/outputs |
| Validation documentation | Test protocols, results, and deviations in evidence pack |
| Revalidation after change | New evidence pack generated post-change |

### 5.3 Change Control Checklist

For each change to the deployed MCP infrastructure:

- [ ] Change category determined (patch, minor, major)
- [ ] Impact assessment completed (conformance levels affected)
- [ ] Regulatory impact assessed (overlay requirements affected)
- [ ] Change implemented in staging environment
- [ ] Full conformance test suite passed
- [ ] New evidence pack generated
- [ ] Change documented in audit ledger
- [ ] Sponsor/CRO notified
- [ ] IRB notified (if protocol-affecting)

### 5.4 Schema Evolution Rules

Per `governance/VERSION_COMPATIBILITY.md`:

- PATCH changes: No schema modifications
- MINOR changes: Additive schema changes only (new optional fields)
- MAJOR changes: Breaking schema changes permitted with 12-month migration period
- Schema diff tool available: `tools/certification/schema_diff.py`

## 6. Inspection Readiness Checklist

### 6.1 Documentation Ready for Inspection

- [ ] Current evidence pack available (< 30 days old)
- [ ] Site capability profile (`deploy/config/site-profile.yaml`)
- [ ] IRB approval documentation
- [ ] Conformance level declaration and supporting test results
- [ ] Regulatory overlay compliance attestation
- [ ] Robot registry with USL scores and certification statuses
- [ ] Staff training records
- [ ] Incident reports and corrective action documentation
- [ ] Data monitoring reports
- [ ] Change control log

### 6.2 System Verification During Inspection

An inspector may request the following live verifications:

| Verification | Tool / Procedure | Expected Result |
|-------------|-----------------|-----------------|
| Chain integrity | `ledger_chain_status` | `chain_valid: true` |
| Full chain verification | `ledger_verify` | `valid: true`, `first_invalid_index: null` |
| RBAC enforcement | `authz_evaluate` with various roles | Matches permission matrix in `spec/actor-model.md` |
| De-identification | `fhir_patient_lookup` | Only pseudonymized data returned |
| Imaging access control | `dicom_query` with restricted role | Appropriate denial or level restriction |
| Token lifecycle | Issue, validate, revoke sequence | Correct behavior at each step |
| Provenance tracking | `provenance_get_lineage` | Complete data lineage visible |
| Safety gate | Review recent gate decisions | All procedure approvals have gate records |

### 6.3 System Access for Inspectors

The `auditor` role (per `spec/actor-model.md`) provides inspection-appropriate access:

| Server | Available Tools |
|--------|----------------|
| trialmcp-authz | `authz_evaluate`, `authz_validate_token`, `authz_list_policies` |
| trialmcp-ledger | `ledger_query`, `ledger_verify`, `ledger_replay`, `ledger_chain_status` |
| trialmcp-provenance | `provenance_get_lineage`, `provenance_get_actor_history`, `provenance_verify_integrity` |

The `auditor` role explicitly CANNOT:
- Access clinical data (`fhir_read`, `fhir_search`, `fhir_patient_lookup`)
- Access imaging data (`dicom_query`, `dicom_retrieve_pointer`)
- Issue or revoke tokens
- Modify audit records

This separation ensures inspectors can verify system integrity and compliance
without accessing patient PHI.

### 6.4 Common Inspection Findings and Mitigations

| Finding | Severity | Mitigation |
|---------|----------|-----------|
| Audit chain break | Critical | Investigate root cause; repair chain; document in CAPA |
| Missing audit entries for tool calls | Major | Verify middleware configuration (`servers/common/middleware.py`) |
| De-identification failure (PHI in logs) | Critical | Review logging configuration (`servers/common/logging.py`) |
| Expired robot certification | Major | Re-certify robot; suspend until recertified |
| Missing safety gate records | Major | Verify gate service integration (`safety/gate_service.py`) |
| Token not revoked after session end | Minor | Review token lifecycle policy |
| Outdated conformance evidence | Minor | Generate current evidence pack |

## 7. Regulatory Submission Guidance

### 7.1 IND/IDE Submissions

For FDA IND (Investigational New Drug) or IDE (Investigational Device Exemption)
submissions that include Physical AI robot procedures:

| Submission Section | MCP Evidence |
|-------------------|-------------|
| Device description | Robot capability descriptors (`schemas/robot-capability-profile.schema.json`) |
| Software description | MCP server architecture (`docs/architecture.md`), conformance level |
| Risk analysis | Safety gate conditions, e-stop procedures, USL scoring |
| Validation documentation | Evidence pack (Section 2) |
| Audit trail capability | Chain architecture description, replay sample |
| Data integrity | Provenance tracking, hash chain verification |
| Access controls | RBAC policy, permission matrix, token lifecycle |

### 7.2 Annual Reports

Include in annual FDA reports:

- Summary of audit chain integrity verifications
- Robot performance metrics (USL score trends)
- Safety gate statistics (pass/fail rates)
- E-stop events and resolutions
- Change control summary
- Updated evidence pack

### 7.3 Safety Reports

For FDA safety reports (21 CFR 312.32 or 812.150):

- E-stop signal records from `safety/estop.py`
- Procedure state history from `safety/procedure_state.py`
- Relevant audit trail via `ledger_replay`
- Provenance chain via `provenance_get_lineage`
- Robot capability and calibration status at time of event

### 7.4 IRB Continuing Review

For IRB continuing review submissions:

| Document | Source |
|----------|--------|
| Protocol compliance summary | Audit trail analysis |
| Adverse event summary | Safety gate failures, e-stop events |
| Enrollment summary | `fhir_study_status` aggregate |
| Data integrity report | Chain verification results |
| System change log | Change control records |
| Updated risk assessment | Safety gate statistics, incident reports |

## 8. Glossary of Regulatory-Technical Terms

| Term | Definition |
|------|------------|
| Audit chain | SHA-256 hash-linked sequence of audit records in the ledger |
| Conformance level | Numbered tier (1-5) indicating which servers and tools are implemented |
| De-identification | Removal or transformation of direct patient identifiers per HIPAA Safe Harbor |
| Evidence pack | Cryptographically sealed bundle of conformance and validation artifacts |
| Gate decision | Aggregate result of safety pre-check evaluation before a procedure |
| Genesis hash | The initial hash in the audit chain (`"0" * 64`) |
| HMAC-SHA256 | Keyed hash used for pseudonymization with site-specific salt |
| Procedure state machine | Enforced lifecycle states for robot procedures |
| Provenance | Recorded lineage of data access, transformation, and aggregation |
| Pseudonymization | Reversible identifier replacement (HMAC-SHA256) for patient IDs |
| RBAC | Role-based access control enforced by the AuthZ server |
| Regulatory overlay | Jurisdiction-specific requirements layered on base conformance |
| Task order | Structured work order for a robot-assisted procedure |
| USL score | Unification Standard Level score measuring robot readiness (0-100) |
