# Sponsor / CRO Implementation Guide

**National MCP-PAI Oncology Trials Standard**
**Audience**: Pharmaceutical Sponsors, Medical Device Companies, Contract Research Organizations
**Version**: 0.1.0

---

## 1. Introduction

This guide provides pharmaceutical sponsors and contract research organizations (CROs)
with operational procedures for managing multi-site Physical AI oncology clinical trials
under the National MCP-PAI standard. It covers cross-site reporting, provenance and
audit review, regulatory overlay implications, multi-site coordination, data quality
monitoring, and study close-out.

Sponsors and CROs are two of the six actor roles defined in `spec/actor-model.md`:

| Role | Trust Level | Access Scope |
|------|-------------|-------------|
| `sponsor` | Governance | Policy configuration, aggregate reporting; no direct patient data access |
| `cro` | Coordination | Cross-site aggregate access; no direct patient-level data |

Both roles operate at the organizational level. Neither role has access to individual
patient PHI through the MCP infrastructure. All patient-level data is pseudonymized
before it reaches sponsor/CRO tools.

## 2. Cross-Site Reporting Interfaces

### 2.1 Available Tools

Sponsors and CROs have access to the following tools across the five MCP servers
(per `spec/actor-model.md` permission matrix):

| Server | Tool | sponsor | cro | Purpose |
|--------|------|:-------:|:---:|---------|
| trialmcp-authz | `authz_evaluate` | ALLOW | ALLOW | Verify access policies |
| trialmcp-authz | `authz_issue_token` | ALLOW | ALLOW | Issue scoped tokens |
| trialmcp-authz | `authz_validate_token` | ALLOW | ALLOW | Validate active tokens |
| trialmcp-authz | `authz_list_policies` | ALLOW | ALLOW | Review RBAC policies |
| trialmcp-authz | `authz_revoke_token` | ALLOW | ALLOW | Revoke compromised tokens |
| trialmcp-fhir | `fhir_study_status` | ALLOW | ALLOW | Enrollment and study status |
| trialmcp-ledger | `ledger_query` | ALLOW | ALLOW | Query audit records |
| trialmcp-ledger | `ledger_replay` | ALLOW | ALLOW | Sequential audit traces |
| trialmcp-ledger | `ledger_chain_status` | ALLOW | ALLOW | Chain integrity status |
| trialmcp-provenance | `provenance_get_lineage` | ALLOW | ALLOW | Data lineage review |
| trialmcp-provenance | `provenance_get_actor_history` | ALLOW | ALLOW | Actor activity review |
| trialmcp-provenance | `provenance_verify_integrity` | ALLOW | ALLOW | Data integrity checks |

Tools explicitly denied to sponsor/CRO roles:

| Tool | Reason for Denial |
|------|-------------------|
| `fhir_read`, `fhir_search`, `fhir_patient_lookup` | Direct patient data access restricted |
| `dicom_query`, `dicom_retrieve_pointer`, `dicom_study_metadata`, `dicom_recist_measurements` | Direct imaging access restricted |
| `ledger_append` | Audit entries generated by system, not by organizational actors |

### 2.2 Cross-Site Study Status Reporting

To obtain aggregate enrollment data across sites:

```python
# Query study status at each participating site
study_status = fhir_study_status(study_id="NCT00000001")
# Returns: study_id, title, status, enrollment_count, phase
```

The federated coordination layer (`integrations/federation/coordinator.py`) aggregates
study status from multiple sites while enforcing data residency policies
(`integrations/federation/policy_enforcement.py`).

### 2.3 Cross-Site Provenance Reporting

Use the provenance tools to trace data lineage across sites:

```python
# Get full lineage for a data source
lineage = provenance_get_lineage(
    source_id="src-clinical-001",
    direction="forward"
)

# Review all operations by a specific actor
actor_history = provenance_get_actor_history(
    actor_id="robot-da-vinci-001",
    start_time="2026-01-01T00:00:00Z",
    end_time="2026-03-31T23:59:59Z"
)
```

Cross-site provenance scenarios are validated in the interop testbed:
`interop-testbed/scenarios/cross_site_provenance.py`

## 3. Provenance and Audit Review Procedures

### 3.1 Audit Chain Integrity Verification

Before reviewing audit data, verify chain integrity at each site:

```python
# Step 1: Check chain status
chain_status = ledger_chain_status()
# Verify: chain_valid == true, genesis_hash == "0" * 64

# Step 2: Full chain verification (periodic)
verification = ledger_verify(start_index=0)
# Verify: valid == true, first_invalid_index == null
```

If `chain_valid` returns `false`, escalate immediately to the site IT team and
document the finding. Chain integrity failures are reportable events under
21 CFR Part 11 (see `profiles/country-us-fda.md` Section 3.5).

### 3.2 Audit Query Procedures

Query audit records filtered by server, tool, caller, or time range:

```python
# Query all robot procedure events at a site
robot_audits = ledger_query(
    server="trialmcp-dicom",
    caller="robot-da-vinci-001",
    start_time="2026-03-01T00:00:00Z",
    end_time="2026-03-08T23:59:59Z"
)

# Query all authorization decisions
authz_audits = ledger_query(
    server="trialmcp-authz",
    tool="authz_evaluate"
)
```

### 3.3 Compliance Replay Traces

Generate sequential replay traces for ICH-GCP E6(R2) compliance review:

```python
# Generate a trace for a specific time window
replay = ledger_replay(
    start_time="2026-03-01T00:00:00Z",
    end_time="2026-03-08T23:59:59Z"
)
# Returns: trace (sequential list), total records, duration

# Generate a trace for a specific actor (e.g., robot)
actor_replay = ledger_replay(
    caller="robot-da-vinci-001",
    start_time="2026-03-01T00:00:00Z",
    end_time="2026-03-08T23:59:59Z"
)
```

Replay traces MUST be sufficient for ICH-GCP E6(R2) audit review (per
`spec/tool-contracts.md` Section 6.4).

### 3.4 Provenance Integrity Verification

Verify that data has not been altered since it was recorded:

```python
# Verify data integrity against recorded fingerprint
result = provenance_verify_integrity(
    source_id="src-clinical-001",
    data="<current data for comparison>"
)
# Returns: verified (bool), expected_hash, actual_hash
```

### 3.5 Routine Review Schedule

| Review Activity | Frequency | Responsible Party |
|----------------|-----------|-------------------|
| Chain integrity verification | Weekly | CRO data monitor |
| Audit query review (authorization decisions) | Weekly | CRO data monitor |
| Provenance lineage review | Monthly | CRO data monitor |
| Actor history review (robot agents) | Per procedure | Trial coordinator |
| Full compliance replay | Quarterly | Sponsor quality team |
| Cross-site aggregate reconciliation | Monthly | CRO coordinator |

## 4. Regulatory Overlay Implications by Jurisdiction

### 4.1 Overview of Regulatory Framework

The standard defines a layered regulatory model:

```
  +------------------------------------------+
  | Federal: 21 CFR Part 11, HIPAA           |
  | profiles/country-us-fda.md               |
  | regulatory/CFR_PART_11.md                |
  | regulatory/HIPAA.md                      |
  +------------------------------------------+
  | State: Jurisdiction-specific overlays     |
  | profiles/state-us-ca.md (CA: CCPA/CPRA)  |
  | profiles/state-us-ny.md (NY: PHL/SHIELD) |
  +------------------------------------------+
  | Site: IRB and institutional policies      |
  | regulatory/IRB_SITE_POLICY_TEMPLATE.md   |
  +------------------------------------------+
```

### 4.2 Federal Requirements (All Sites)

All sites MUST comply with:

| Regulation | Key Requirements | MCP Implementation |
|-----------|-----------------|-------------------|
| 21 CFR Part 11 | Audit trails, electronic signatures, system validation | Hash-chained ledger, token-based auth, conformance testing |
| HIPAA Privacy Rule | PHI de-identification, minimum necessary | HMAC-SHA256 pseudonymization, role-based access |
| HIPAA Security Rule | Access controls, audit controls, integrity | RBAC, audit ledger, chain verification |
| ICH-GCP E6(R2) | Source data verification, audit trail | Provenance tracking, replay traces |

Reference: `regulatory/US_FDA.md`, `regulatory/CFR_PART_11.md`, `regulatory/HIPAA.md`

### 4.3 California Sites (CCPA/CPRA)

Additional requirements for California-based sites (per `profiles/state-us-ca.md`):

| Requirement | Impact on Sponsor/CRO |
|-------------|----------------------|
| Data subject access requests | MUST support consumer request fulfillment for non-PHI data |
| Right to delete | MUST accommodate deletion requests for CCPA-covered data |
| Data processing agreements | MUST execute CPRA-compliant service provider agreements with each CA site |
| Privacy impact assessments | MUST conduct PIA for robot telemetry linked to patients |
| CCPA-HIPAA boundary | Pseudonymized data (reversible) MUST be treated as personal information |

### 4.4 New York Sites (PHL/SHIELD Act)

Additional requirements for New York-based sites (per `profiles/state-us-ny.md`):

| Requirement | Impact on Sponsor/CRO |
|-------------|----------------------|
| HIV-related data segregation | FHIR resources with HIV data require separate access authorization |
| PHL Article 27-F | Written authorization required for HIV data disclosure |
| SHIELD Act breach notification | 72-hour notification requirement for security breaches |
| NY DOH research protections | Additional reporting for research participant protections |

### 4.5 Multi-Jurisdiction Trial Considerations

For trials spanning multiple states:

1. **Identify applicable overlays**: Each site applies its own state overlay in addition to federal requirements
2. **Configure site profiles**: Each site's `jurisdiction` block declares applicable requirements (see `deploy/config/site-profile-example.yaml`)
3. **Harmonize policies**: The CRO MUST maintain a jurisdiction matrix documenting overlay requirements per site
4. **Federated data handling**: The federation policy enforcement layer (`integrations/federation/policy_enforcement.py`) prevents data residency violations
5. **Privacy budget**: Track aggregate privacy exposure across sites (`integrations/privacy/privacy_budget.py`)

### 4.6 Jurisdiction Decision Matrix

| Requirement | Federal | California | New York |
|-------------|:-------:|:----------:|:--------:|
| Audit trail retention | 21 CFR 11.10(e) | CCPA 1798.105(d) | PHL 18 |
| Breach notification | 60 days (HIPAA) | 72 hours (CCPA) | "Most expedient" (SHIELD) |
| Right to access | HIPAA access right | CCPA consumer right | PHL patient right |
| De-identification standard | HIPAA Safe Harbor | CCPA threshold | HIPAA + PHL stricter |
| Consent model | ICH-GCP | CCPA opt-out | PHL written consent |

## 5. Multi-Site Deployment Coordination

### 5.1 Site Onboarding Process

The CRO coordinates site onboarding using the following process:

1. **Site qualification**: Verify infrastructure requirements (see `docs/guides/hospital-it.md`)
2. **Profile configuration**: Assist site IT with `deploy/config/site-profile-example.yaml` customization
3. **Conformance level alignment**: Ensure all sites target the same conformance level
4. **Regulatory overlay identification**: Map each site to applicable state overlay
5. **Identity federation**: Configure cross-site token exchange (`interop-testbed/scenarios/token_exchange.py`)
6. **Connectivity testing**: Run cross-site communication tests (`interop-testbed/scenarios/site_onboarding.py`)
7. **Evidence pack generation**: Each site generates certification evidence (`tools/certification/evidence_pack.py`)

### 5.2 Version Compatibility Across Sites

Per `governance/VERSION_COMPATIBILITY.md`:

| Rule | Requirement |
|------|-------------|
| Same MAJOR version | MUST across all sites in a trial |
| Same MINOR version | SHOULD across all sites in a trial |
| PATCH differences | Acceptable; no action required |
| Upgrade coordination | MINOR upgrades within 6 months of release |
| MAJOR migration | 12 months allowed; migration guide required |

### 5.3 Federated Data Governance

```
  Site A ──┐
           ├── Secure Aggregation ── Aggregate Results ── Sponsor Dashboard
  Site B ──┤   (integrations/federation/secure_aggregation.py)
           │
  Site C ──┘
```

Key modules:
- `integrations/federation/coordinator.py` -- cross-site orchestration
- `integrations/federation/secure_aggregation.py` -- privacy-preserving aggregation
- `integrations/federation/site_harmonization.py` -- data harmonization across sites
- `integrations/federation/policy_enforcement.py` -- data residency enforcement

### 5.4 Site Harmonization

The site harmonization module (`integrations/federation/site_harmonization.py`) handles:

- Terminology normalization across sites
- Schema version differences
- Time zone standardization
- Identifier namespace resolution

## 6. Data Quality Monitoring

### 6.1 Automated Quality Checks

| Check | Method | Frequency |
|-------|--------|-----------|
| Schema compliance | Validate against `schemas/*.schema.json` | Every transaction |
| Audit chain integrity | `ledger_verify` | Weekly |
| Provenance completeness | `provenance_get_lineage` traversal | Weekly |
| Cross-site consistency | `fhir_study_status` reconciliation | Weekly |
| Robot performance | Benchmark suite (`benchmarks/`) | Monthly |
| Safety gate compliance | Gate decision audit review | Per procedure |

### 6.2 Data Monitor Role

The `data_monitor` role (per `spec/actor-model.md`) has read-only access to clinical
and imaging data for quality review. Data monitors can:

| Capability | Tool | Access Level |
|-----------|------|-------------|
| Query clinical data | `fhir_read`, `fhir_search` | Read-only, de-identified |
| Review patient status | `fhir_patient_lookup` | Pseudonymized only |
| Query imaging metadata | `dicom_query` | PATIENT/STUDY level only |
| Review audit records | `ledger_query`, `ledger_replay` | Full access |
| Check provenance | `provenance_get_lineage`, `provenance_get_actor_history` | Full access |

Data monitors CANNOT:
- Issue or revoke tokens
- Append to the audit ledger
- Retrieve DICOM images
- Access SERIES or IMAGE level DICOM queries

### 6.3 Quality Event Escalation

| Severity | Examples | Escalation Path | Timeline |
|----------|---------|-----------------|----------|
| Critical | Chain integrity failure, safety gate bypass | Sponsor safety + site PI + IRB | Immediate |
| High | Protocol deviation, missing audit entries | Sponsor medical monitor + site coordinator | 24 hours |
| Medium | Data inconsistency, late data entry | CRO data manager + site coordinator | 1 week |
| Low | Minor formatting issues, non-critical gaps | CRO data manager | Next monitoring visit |

## 7. Study Close-Out Procedures

### 7.1 Pre-Close-Out Checklist

- [ ] All enrolled subjects completed or withdrawn per protocol
- [ ] All robot procedures completed with final status (`completed`, `cancelled`, or `failed`)
- [ ] All safety events reported and resolved
- [ ] All data queries resolved at each site
- [ ] Final data quality review completed

### 7.2 Data Lock Procedures

1. **Freeze audit chain**: Record study close-out in ledger at each site
2. **Export final audit chain**: Run `ledger_replay` for the full study duration at each site
3. **Verify chain integrity**: Run `ledger_verify` across full chain at each site
4. **Export provenance records**: Run `provenance_get_lineage` for all registered sources
5. **Generate final evidence packs**: `tools/certification/evidence_pack.py` at each site
6. **Archive site profiles**: Preserve final `site-profile.yaml` for each site

### 7.3 Regulatory Archive Package

Assemble the following for regulatory archive:

| Component | Source | Format |
|-----------|--------|--------|
| Complete audit trail | `ledger_replay` (full study) | JSON + human-readable export |
| Provenance graph | `provenance_get_lineage` (all sources) | JSON |
| Conformance evidence | `tools/certification/evidence_pack.py` | Evidence bundle |
| Site profiles | `deploy/config/site-profile.yaml` per site | YAML |
| Robot registry snapshots | `safety/robot_registry.py` exports | JSON |
| Safety gate decisions | Gate decision audit records | JSON |
| Schema versions | `schemas/*.schema.json` as deployed | JSON Schema |
| Specification version | `spec/` as deployed | Markdown |

### 7.4 Record Retention

Per 21 CFR Part 11 and ICH-GCP requirements:

| Record Type | Retention Period |
|------------|-----------------|
| Audit ledger | Duration specified by FDA predicate rule (minimum 2 years post-approval) |
| Provenance records | Co-terminus with audit ledger |
| Evidence packs | Co-terminus with audit ledger |
| Robot certification records | Duration of trial + 3 years |
| Site profiles | Duration of trial + 3 years |
| Incident reports | Duration of trial + 3 years minimum |

### 7.5 Site Decommissioning

After archive completion:

1. Confirm all data exported and archived
2. Revoke all active tokens at the site
3. Verify no active robot agents registered
4. Decommission MCP server instances
5. Secure-erase any local PHI per institutional policy
6. Document decommissioning in sponsor's TMF (Trial Master File)
