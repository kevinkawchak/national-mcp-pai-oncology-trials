Abstract

Prior oncology clinical trial infrastructure has been fragmented, site-specific, and inefficient across the critical dimensions of interoperability, auditability, privacy, and deployment. Each trial site has historically operated its own isolated systems for electronic health records, imaging archives, and audit logging, resulting in inconsistent data formats, duplicated regulatory effort, and limited cross-site collaboration.

This page presents a demonstration of the National MCP Physical AI Oncology Trials system, a proposed end-to-end architecture comprising five Model Context Protocol (MCP) servers that address these gaps through standardized, federated, and safety-governed capabilities. The five servers (Authorization, FHIR Clinical Data, DICOM Imaging, Audit Ledger, and Provenance) expose 23 tools across five hierarchical conformance levels, validated by 668 test functions.

Hash-Chained Audit Record (from servers/trialmcp_ledger/chain.py)

GENESIS_HASH = "0" * 64

def canonical_json(record: dict[str, Any]) -> str:
    filtered = {k: v for k, v in sorted(record.items())
                if k != "hash"}
    return json.dumps(filtered, sort_keys=True,
                      ensure_ascii=True)

def compute_audit_hash(record, prev_hash: str) -> str:
    canonical = canonical_json(record)
    payload = prev_hash + canonical
    return hashlib.sha256(
        payload.encode("utf-8")).hexdigest()

Cross-site token exchange enables authorized actors to access resources across sites without requiring site-specific credentials. 4 Safety Gate Evaluation

Source: docs/mcp-process/04-safety-gate-evaluation.md

Five mandatory pre-procedure safety gates must pass sequentially before any robotic procedure can proceed. Each gate has a 300-second timeout. If any gate fails, the procedure transitions to ABORTED and a full audit record is created.

1
Patient Consent
Verify active, informed consent for the specific procedure
PENDING
2
Site Capability
Validate site meets required conformance level
PENDING
3
Robot Capability
Confirm robot firmware and capability profile match task requirements
PENDING
4
Protocol Compliance
Verify procedure parameters satisfy trial protocol constraints
PENDING
5
Human Approval
Require explicit clinician authorization (300s timeout)
PENDING

The gate evaluation animation above demonstrates the sequential pass/fail workflow. In production, the gate service (safety/gate_service.py) integrates with all five MCP servers to validate each condition.

5 Federated Learning Coordination

Source: docs/mcp-process/05-federated-learning-coordination.md

Multi-site federated learning enables collaborative model training without centralizing patient data. The system supports three aggregation strategies with differential privacy budget tracking per site.

Aggregation Strategies

  1. FedAvg: Weighted averaging proportional to local dataset size
  2. FedProx: Proximal term regularization for heterogeneous data
  3. SCAFFOLD: Variance reduction through control variates

Privacy Controls

  • Differential privacy budget (epsilon) tracking per site per round
  • Secure aggregation with mask generation
  • Privacy adapter: integrations/privacy/privacy_budget.py
1 Coordinator distributes global model to sites
2 Each site trains locally on private data
3 Sites generate masks for secure aggregation
4 Masked model updates submitted to coordinator
5 Aggregation (masks cancel), global model updated
6 Updated model distributed, privacy budgets decremented

Federation adapters are implemented in integrations/federation/, including the coordinator (coordinator.py), policy enforcement (policy_enforcer.py), secure aggregation (secure_aggregation.py), and site harmonization (site_harmonization.py) modules.

6 Audit and Provenance Chain

Source: docs/mcp-process/06-audit-provenance-chain.md

7 Privacy and De-identification

Source: docs/mcp-process/07-privacy-deidentification.md

The HIPAA Safe Harbor 18-identifier pipeline removes protected health information (PHI) across three stages before clinical data leaves the server boundary. HMAC-SHA256 pseudonymization preserves referential integrity across de-identified records.

  • Biometric identifiers

Data residency enforcement ensures compliance with federal, California (CCPA/CMIA), and New York state privacy regulations through the data residency adapter (integrations/privacy/data_residency.py). Access control combines RBAC and ABAC checks across four data classification levels.

Emergency Stop Code (from safety/estop.py)

class EStopStatus(Enum):
    IDLE = "IDLE"
    TRIGGERED = "TRIGGERED"
    ACKNOWLEDGED = "ACKNOWLEDGED"
    RECOVERING = "RECOVERING"

class EStopController:
    def trigger_estop(self, procedure_id, reason,
                      triggered_by, current_state=None):
        if self._status == EStopStatus.TRIGGERED:
            raise RuntimeError("E-stop already active")
        signal = EStopSignal(
            signal_id=str(uuid.uuid4()),
            procedure_id=procedure_id,
            reason=reason,
            triggered_by=triggered_by,
            affected_servers=list(self._active_servers),
            preserved_state=current_state or {})
        self._active_signal = signal
        self._status = EStopStatus.TRIGGERED
        return signal

Adoption Roadmap

Stakeholder-specific pathways for implementing the national standard

S

Sponsors and CROs

Begin with Level 1 (Core) deployment for centralized audit trail access. Progress to Level 2 for de-identified clinical data queries. Guide: docs/guides/sponsor-cro.md

H

Hospital IT

Deploy five-server stack using Docker Compose (deploy/docker-compose.yml). Integrate with existing EHR via FHIR adapters and PACS via DICOM adapters. Guide: docs/guides/hospital-it.md

R

Regulators and IRBs

Review conformance profiles and regulatory overlays (profiles/). Evaluate audit chain integrity using ledger_verify. Guide: docs/guides/regulator-irb.md

V

Robot Vendors

Implement MCP client interface against the 23-tool contract. Register robots via safety/robot_registry.py. Run conformance tests at Level 5. Guide: docs/guides/robot-vendor.md

C

Standards Community

Review the 9 normative specifications in spec/, 7 ADRs in docs/adr/, and governance framework. Guide: docs/guides/standards-community.md

Classification and Status

What This IS

A reference architecture demonstration and interactive simulation of the proposed National MCP-PAI Oncology Trials Standard. It serves as an educational and stakeholder communication tool that visualizes the system described in the paper using deterministic simulations with pre-configured data.

What This Is NOT

  • NOT a deployed clinical system
  • NOT approved for use in actual oncology trials
  • NOT connected to real patient data, EHR systems, PACS, or robotic systems
  • NOT a medical device or software as a medical device (SaMD)

Simulation Conditions and Approximations

  • All data shown is synthetic
  • Network latencies, site counts, and response times are modeled approximations based on the paper's architecture
  • Federated learning rounds use illustrative aggregation, not real model weights
  • Hash chains demonstrate the algorithm with sample data

What Is Needed for Actual Trial Use

  1. FDA review and recognition as a consensus standard
  2. Clinical pilot deployment at NCI-designated cancer centers
  3. Real EHR/PACS adapter validation with vendor systems (Epic, Cerner, Orthanc, dcm4chee)
  4. Robot vendor certification program
  5. IRB approval at each participating site
  6. 21 CFR Part 11 compliance audit of deployed instances
  7. HIPAA security risk assessment

Current Status

This is a proposed reference standard with a complete reference implementation (381 files, approximately 69,800 lines) that has not yet been deployed in clinical settings. The repository (v1.2.0) includes the complete specification, reference implementations in Python and TypeScript, a 668-function test suite, and a 20-page peer-reviewed paper.
